# ADR-014: Knowledge Network Transport and Sync Model

**Status:** Proposed
**Date:** 2026-06-25
**Milestone:** 2 (Cross-repo knowledge network GA)

## Context

v0.2 takes the publish/import pipeline out of advisory mode into a real fetch path across a 6–100 repo single-org estate. The owner recommended two options (polling-first with optional webhooks) and rejected MCP-server-only (not CI-portable) and P2P mesh (overkill for a single org). A red-team pass before the v0.2 architecture was finalised identified two load-bearing problems: (a) timestamp handling was one-sided in the attacker's favour (`expires_at` could keep a withdrawn packet adoptable past any receiver-side horizon), and (b) the SSRF guard in the draft `poll-network.mjs` relied on regex-on-the-URL rather than on the resolved IP, leaving metadata-endpoint redirects and DNS-rebinding open. This ADR fixes the transport model, the freshness model, and the SSRF defence. It operates within the CI boundary and CODEOWNERS boundary established in `ARCHITECTURE.md` as structural security invariants, and within the ADR-010 trust model (local gates + independent checker + owner decision required for any adoption).

The network feature ships OFF BY DEFAULT: `repo_network_enabled` unset; `.modonome/NETWORK.md` `Mode: disabled`. Enabling it is Tier 4 and requires an explicit owner decision.

## Decision

**1. Pull-only is the trust boundary.** `scripts/poll-network.mjs` makes only outbound HTTPS GETs to an allowlisted, signed catalog origin (Tier 4 to change). No inbound ports are opened; no listener ships. This is why polling suits a firewalled corporate estate and why the two rejected alternatives fail: MCP-server-only is not CI-portable and therefore cannot satisfy the ARCHITECTURE.md invariant that modonome "needs no central service"; P2P mesh introduces a routing authority and peer-discovery surface that the sovereignty model forbids for a single-org estate. A first-class script (not raw `git fetch`) is kept because id-recompute, signature verification, expiry, atomic staging, and SSRF defence all live in it. (A bare `git fetch` provides none of those properties.)

**2. Poll is the source of truth; freshness is never a fast path; `expires_at` may only shorten validity.** Staleness, replay, and out-of-order delivery cannot cause a wrong adoption because adoption requires local gates (ADR-010) regardless. The result is a missed opportunity, never a wrong outcome. `published_at`, `expires_at`, and catalog `generated_at` are therefore advisory metadata, never trust inputs, and all are pinned to RFC3339 UTC. Ordering uses `lineage.supersedes_packet_ids`, never timestamps. Because `expires_at` is attacker-controlled it may only shorten validity: the receiver enforces its own `network_catalog_max_age_hours` cap independent of the packet's claim. The `network_clock_skew_seconds` tolerance (default 300) is applied conservatively only: a packet ambiguous at the boundary is treated as expired, never as fresh. This closes red-team finding H5.

**3. The network poll runs in CI scope from the protected base branch (ADR-019).** `node scripts/poll-network.mjs` runs in any CI (Actions, Jenkins, GitLab) with only network egress, matching the ARCHITECTURE.md invariant. Critically, it joins the same base-branch execution trust class as `guard-ratchet.mjs`: a scheduled CI job, never an agent-turn invocation, never a PR-head checkout. `scripts/poll-network.mjs`, `scripts/verify-packet.mjs`, `scripts/validate-knowledge-packet.mjs`, and every `scripts/lib/*` they load run from the protected base branch; the CI job refuses to start on a PR head and self-hash-guards each network script against its base-branch hash. Inbox status (`pending-local-validation`) and the audit ledger (`.modonome/network/ledger.jsonl`) are written only by this CI job; the agent may read them but cannot forge them. A scheduled nightly/per-cycle poll replaces any daemon. Without ADR-019 the signing, identity, and redaction stack is attacker-controlled theater.

**4. SSRF defence is resolved-IP, not regex-on-URL (red-team H10).** `scripts/poll-network.mjs` disables redirect-following entirely. It resolves the catalog hostname to an IP and validates the resolved IP (not the hostname string) against a denylist that includes loopback (`127.0.0.0/8`), all RFC1918 ranges, link-local `169.254.0.0/16`, and IPv6 ULA/link-local. This closes metadata-endpoint redirects and DNS-rebinding at the network layer. TLS verification is never disabled. Response size and timeout are capped. The SSRF guard is implemented in `scripts/lib/net-guard.mjs` and does not depend on the redaction regexes in `scripts/validate-knowledge-packet.mjs`.

**5. Webhooks are deferred to v0.3.** An inbound listener inverts the pull-only trust direction and adds request-auth, replay, DoS, and a pre-validation parse surface (a permanent Tier 3+ protected-path maintenance burden) for zero capability gain over polling: poll already delivers every packet; webhooks would buy latency reduction only. The trigger to revisit is a measured sub-poll-cycle propagation latency need that an estate cron cannot meet. If ever built, it must be a non-authoritative hint that re-enters the exact base-branch `poll-network.mjs` verify path, writing only to `.modonome/network/webhook-inbox/`, never to the staging surface directly.

**6. Fail-closed-to-no-adoption.** Catalog unreachable → `exit 0` plus a warning; import nothing; a missed cycle is harmless. `--strict` produces a non-zero exit for pipelines that require it. An invalid index signature or a sequence-rollback (monotonic `sequence` in `schemas/catalog-index.schema.json`, enforced against the CI-written last-seen sequence) aborts the whole poll, not the offending packet alone, because a rolled-back index may be withholding a security fix. Per-packet quarantine isolates a single bad body from the rest of the batch. Atomic temp-file-plus-rename staging mirrors ADR-007: a partial write is never visible as a staged packet.

## Consequences

- The network can never block or regress a host repo's own pipeline; catalog unreachability is a warning, not a failure, unless `--strict` is set.
- Receivers expose no inbound attack surface in v0.2; the only network activity is an outbound HTTPS GET from a scheduled CI job.
- A far-future `expires_at` cannot keep a withdrawn packet adoptable past the receiver's own `network_catalog_max_age_hours` cap; the attacker-favorable expiry gap (red-team H5) is closed at the receiver.
- DNS-rebinding and metadata-endpoint SSRF are closed at the resolved-IP layer in `scripts/lib/net-guard.mjs`, independent of the redaction regex stack (red-team H10).
- Transient catalog outages degrade gracefully; a sequence-rollback or index-signature failure is a hard abort to prevent silent withholding of a security fix.
- Enabling network behavior, changing the catalog-origin allowlist, or enrolling a peer key remains Tier 4 (owner decision only); the agent cannot write `Mode:`, `peer-keys.json`, or any CI-written ledger.
- The base-branch execution requirement (ADR-019) is a hard precondition: any `poll-network.mjs` invocation outside CI scope on the protected base branch produces no staging output and exits non-zero; skipping it voids every other mitigation.
- Referenced scripts (`scripts/poll-network.mjs`, `scripts/lib/net-guard.mjs`, `scripts/verify-packet.mjs`, `scripts/validate-knowledge-packet.mjs`) and schemas (`schemas/catalog-index.schema.json`) are protected paths (>= Tier 3 via CODEOWNERS); weakening any of them requires CODEOWNERS approval and is therefore visible in review history.
