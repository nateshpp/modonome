The problem

Autonomous agents weaken gates to go green

Coding agents have a predictable failure mode. Told to make tests pass, they remove assertions, add skips, and loosen type checks. Modonome blocks that structurally. The anti-gaming ratchet runs in CI from a base-branch copy the agent's run does not control, and it rejects any diff that weakens a gate.

How it works

Adopt, propose, make, check, gate, merge, learn

Maker, checker, and merge authority are structurally separate. A human owner stays in control of every lever.

  1. Adopt. Read the host repo's instructions, CI, code owners, gates, and conventions, then defer to them.
  2. Dry-run. Propose bounded work as a queue, read-only. It writes nothing until you arm it.
  3. Make. A maker implements one tightly scoped packet, with a failing test as the fence.
  4. Check. An independent checker, separate from the maker, runs the gates and reviews the diff.
  5. Gate. Deterministic gates and the anti-gaming ratchet run in CI, outside the agent's reach.
  6. Owner. Protected paths (CI, secrets, schemas, migrations, auth) wait for a human decision.
  7. Merge. A separate merge authority lands the change only when every gate is green.
  8. Learn. Real corrections become staged lessons that an owner promotes into durable rules.

Why it is safe to run

The controls live in CI, not in the prompt

A prompt can be overridden by a cleverer prompt. A CI gate that runs outside the agent's write scope holds. The arming levers live in environment variables, outside the agent's read scope. With MODONOME_ARMED unset, autonomy is forced off no matter what any config file says.

What you get

Structural controls, not promises

{ }

Anti-gaming ratchet

Rejects diffs that remove assertions, drop coverage, add skips, or loosen type safety. Runs from a trusted base-branch copy.

II

Independent checker

The role that verifies a change is structurally separate from the role that wrote it, and from the one that merges it.

⏻

Off by default

Autonomy and auto-merge stay off until an owner arms them through CI or environment, never through a file the engine can rewrite.

☑

Bounded work items

Tier 1 tests and docs, Tier 2 scripts and schemas. Small, dated, capped, and test-fenced. Each PR stays reviewable.

⚠

Protected paths

CI, secrets, schemas, migrations, lockfiles, and auth always wait for owner review through your existing CODEOWNERS.

↻

Learns with a human in the loop

Gate failures and reversions become staged lessons. An owner promotes durable ones into canonical rules and new gates.

☷

Stack detection

Detects Node, Python, Java, and .NET and their gates on adoption. Enterprise mirror mode is on the roadmap for non-Git estates.

$ npx modonome scaffold . --write    # apply them (still off)
